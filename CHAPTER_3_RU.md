# Глава 3 — полноценная система тестирования

Глава закрывает риск, при котором отдельные самопроверки проходили, а реальный
пользовательский сценарий мог оставаться сломанным. Рабочий Telegram-бот
проверяется без сетевых запросов и без изменения настоящих файлов состояния.
Архивные Mini App, Worker и D1 в эту главу не входят.

## Что теперь проверяется

1. Изолированное тестовое окружение с тестовыми ключами и временными файлами.
2. Единый набор `pytest`, который запускается на каждом push в `main`, каждом PR
   и вручную.
3. Матрица кнопок для пользователя и администратора: участие, скрытие,
   «Неактивное», завершение и установка времени.
4. Полный путь уведомления от формирования найденного колеса до вызова Telegram.
5. Два канала с одним колесом дают одно уведомление каждому получателю.
6. Повторно использованный `/freestream/<id>` считается новым событием после
   завершения предыдущего и окончания периода дедупликации.
7. Одновременные доставка, регистрация, личная отметка, заявка и запись
   административной команды не затирают друг друга.
8. Эмулируются таймаут Telegram, повреждённый журнал доставки, ошибка действия
   и конфликт GitHub при записи очереди.
9. CI сверяет `HEAD` с `GITHUB_SHA`, то есть тестирует именно вызвавший его commit.
10. Для критического ядра уведомлений, дедупликации, публикаций и рейтинга
    установлен обязательный порог покрытия 80%.

## Важное изменение CI

Старый workflow мог скрыть падение через `continue-on-error`, после чего сам
записывал результат в `main`. Теперь проверка имеет только `contents: read`,
любая ошибка завершает job с ошибкой, а устаревающий
`current_validation_state.json` удалён.

## Дополнительное исправление, найденное тестами

Между проверкой повтора и отметкой успешной доставки было небольшое окно гонки.
При двух одновременных потоках одно колесо могло уйти дважды. Теперь сообщение
сначала атомарно резервируется: второй поток его пропускает, успешная отправка
попадает в постоянный HMAC-журнал, а неудачная освобождает резерв для повтора.
