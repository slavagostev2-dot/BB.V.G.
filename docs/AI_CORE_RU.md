# BB V.G. — AI Core и интеллектуальные функции

## Назначение

`bbvg/ai_core.py` — единый необязательный шлюз для AI-функций проекта.
Он не владеет обнаружением колёс, Telegram-уведомлениями, автоучастием или
жизненным циклом колёс и не должен становиться обязательной зависимостью этих
контуров.

Главный принцип: отказ внешнего AI не должен останавливать ни один критический
контур BB V.G. Там, где возможно безопасное точное правило, оно имеет приоритет
над AI. AI используется как дополнительный классификатор или объясняющий слой.

## Главный контракт безопасности

Если AI выключен, не настроен, превысил лимит, вернул ошибку или некорректный
JSON, вызывающий код получает детерминированный fallback. Исключение из AI Core
не должно останавливать production-монитор или Telegram control center.

Feature flag сам по себе не даёт AI право изменять production-state. Любая
изменяющая команда проходит отдельную предметную валидацию и, где предусмотрено,
явное подтверждение пользователя.

## Feature flags

- `health_inspector` — инспектор здоровья системы;
- `suspicious_post_analysis` — анализ сомнительных Telegram-публикаций;
- `source_discovery` — зарезервированный AI-слой оценки кандидатов источников;
- `natural_language_admin` — разбор администраторских команд на естественном языке.

Глобальное включение: `BBVG_AI_ENABLED=true`.

Список функций задаётся через `BBVG_AI_FEATURES`, например:
`BBVG_AI_FEATURES=health_inspector,natural_language_admin`.
Также поддерживаются индивидуальные переменные вида
`BBVG_AI_FEATURE_HEALTH_INSPECTOR=true`.

## Провайдер

Начальный провайдер — OpenAI Responses API через HTTPS. Ключ читается только из
`OPENAI_API_KEY`, модель — из `BBVG_AI_MODEL`. Ключ никогда не записывается в
runtime-state, журнал решений или логи AI Core.

Если ключ или модель не заданы, AI Core возвращает `not_configured` и fallback.

## Кэш и лимит запросов

Кэш предназначен для одинаковых AI-запросов и ограничивает лишние обращения к
провайдеру. Ключ кэша — SHA-256 от feature, провайдера, модели и полного запроса.
Исходный prompt в журнал решений не записывается.

По умолчанию TTL кэша — 900 секунд (`BBVG_AI_CACHE_TTL_SECONDS`). Общий минутный
лимит — 10 новых запросов (`BBVG_AI_MAX_CALLS_PER_MINUTE`). Попадание в кэш лимит
не расходует. При превышении лимита возвращается `rate_limited` и fallback.

## Журнал решений

`ai_runtime_state.json` хранит ограниченный журнал AI Core и кэш. По умолчанию
сохраняется не более 500 решений.

Журнал содержит ID решения, время, feature, статус, провайдера, модель, факт
использования кэша, SHA-256 результата и безопасные структурированные поля
`decision`, `classification`, `action`, `reason`, `confidence`, если они есть.
Исходный prompt намеренно не сохраняется.

## Structured JSON

`AIClient.ask_json()` требует один JSON-объект. Если ответ не разбирается как
объект, вызывающий код получает `invalid_json` и переданный fallback-объект.
Валидный JSON сам по себе не считается доверенным: каждый предметный модуль
проверяет допустимые enum, диапазоны и параметры на своей границе.

## Пункт 10 — инспектор здоровья системы

Предметный владелец: `bbvg/health_inspector.py`.

Инспектор подключён к централизованным системным проверкам. Строгие правила
`system_checks` остаются единственным источником уровня `ok`, `warning` или
`critical`. AI может только сформулировать понятное объяснение уже вычисленного
состояния.

Инспектор работает read-only: не перезапускает workflow, не изменяет источники,
не исправляет состояние и не выполняет эксплуатационные команды. При отсутствии
AI используется rules-only объяснение. Результат сохраняется в
`system_check_state.json` в секции `ai_health_inspector`.

## Пункт 3 — анализ сомнительных публикаций

Предметный владелец: `bbvg/monitor/suspicious_posts.py`.

Анализ подключён на границе Telegram ingestion через `telegram_post_links_v2.py`.
Он рассматривает только свежие публичные публикации, в которых обычный парсер не
нашёл прямую ссылку `betboom.ru/freestream/...`.

Перед AI работает локальный дешёвый candidate gate по словам о колесе,
прокрутке, BetBoom, розыгрыше и времени события. Обычные посты провайдеру не
передаются.

Допустимые классы:

- `active_wheel`;
- `possible_wheel_announcement`;
- `old_wheel_reference`;
- `betboom_promotion`;
- `irrelevant`;
- `uncertain`.

Даже `active_wheel` является только предположением по тексту. AI не создаёт
активное колесо, не запускает автоучастие, не меняет дедупликацию и не обходит
BetBoom API. Реальное событие по-прежнему проходит обычный технический контур
после появления ссылки.

При высокой уверенности формируется только служебный административный сигнал.
По умолчанию пороги: 0.88 для `active_wheel` и 0.80 для
`possible_wheel_announcement`. Повтор неизменённого поста не создаёт новый
AI-запрос и не дублирует уже доставленное уведомление.

Минимальная история классификаций хранится в существующем `source_stats.json` в
namespace `ai_post_analysis`. Исходный текст поста туда не записывается: хранятся
hash текста, источник, ID/URL публичного сообщения, классификация, confidence,
причина и ID решения.

Любое исключение анализатора перехватывается. Обёртка сохраняет установленный
production-контракт `fetch_all_sources.__module__ == "telegram_transport"`.

Пока AI-провайдер не настроен для monitor workflow, этот слой безопасно остаётся
неактивным и не меняет обычную проверку колёс.

## Пункт 4 — жизненный цикл и рекомендации новых источников

Предметный владелец: `bbvg/monitor/source_discovery.py` поверх существующей
разведки `source_intelligence.py`.

Новая система не создаёт второй сканер. Она оценивает уже накопленные разведкой
факты и присваивает кандидату один из статусов:

- `candidate` — доказательств пока мало;
- `observed` — канал подтверждён и тематически релевантен, продолжается наблюдение;
- `recommended` — накоплены прямые или комбинированные доказательства полезности;
- `approved` — источник уже утверждён в мониторинге;
- `rejected` — источник отклонён администратором.

Рекомендация строится детерминированно по публичности, статусу проверки, рейтингу,
числу найденных ссылок на колёса, независимым упоминаниям и источникам обнаружения.
AI-флаг `source_discovery` зарезервирован для будущего дополнительного слоя, но
текущее production-решение о рекомендации от внешнего AI не зависит.

Только статус `recommended` создаёт карточку администратору. Кнопки позволяют
добавить канал в основной или ночной мониторинг, продолжить наблюдение или
игнорировать. Автоматического бесконтрольного добавления источников нет.

Жизненный цикл хранится внутри существующего `intelligence_state.json`. На первом
production-прогоне были классифицированы уже накопленные кандидаты; отсутствие
`recommended` означает отсутствие кандидатов, достигших строгого порога, а не
ошибку процесса.

## Пункт 7 — профиль охотника за колёсами

Предметный владелец: `bbvg/bot/profile.py`.

Профиль добавлен в Telegram control center отдельной строкой внизу существующего
меню без перестановки прежних кнопок. Он пересчитывает статистику из реальных
сохранённых событий, а не поддерживает вручную увеличиваемый счётчик.

Основной источник истории — `source_stats.json.personal_wheel_votes`, где записи
связаны с конкретным поколением события и псевдонимизированным actor token.
Для владельца дополнительно учитываются подтверждённые browser auto-participation
события со статусами `participated` и `already_participating`.

Профиль показывает:

- общее число уникальных сохранённых событий участия;
- ручные отметки;
- для владельца подтверждённые автоучастия;
- число текущих активных событий с участием;
- текущую и лучшую серию дней;
- лучший месяц;
- число дней в боте;
- достижения по порогам участия, автоучастия и сериям.

Идентичности `#action`, `#generation` и browser `#event` нормализуются для защиты
от двойного подсчёта одного события. Профиль является только presentation layer и
не изменяет жизненный цикл колёс или голосование.

## Пункт 12 — управление естественным языком

Предметный владелец: `bbvg/bot/natural_language_admin.py`.

Слой подключается только к Telegram control center и только для роли owner/admin.
Обычные пользователи, slash-команды и активные pending-input сценарии проходят по
прежним маршрутам.

### Read-only действия

Могут выполняться сразу после уверенного разбора:

- показать активные колёса;
- показать состояние системы;
- показать рейтинг источников;
- показать давно неактивные источники;
- показать список или карточку источника;
- показать личный профиль.

### Изменяющие действия

Поддерживаются только через отдельную кнопку подтверждения:

- изменение интервала проверки на разрешённое значение 1/3/5/10/15/30 минут;
- перевод источника в основной или ночной режим;
- удаление источника из мониторинга.

До подтверждения production-state не меняется. При добавлении или переносе
источника повторно проверяется публичная доступность Telegram-канала.

### Запрещённые через AI операции

Через естественный язык не выполняются управление секретами, передача владельца,
массовое удаление пользовательских данных, переписывание Git-истории, удаление
backup и другие неизвестные критические операции.

Для частых русскоязычных команд реализован детерминированный parser, поэтому
основные команды чтения, интервал и управление источниками работают независимо
от внешнего AI. Если команда не покрыта строгими правилами, может использоваться
AI Core с allowlist intent и порогом confidence 0.72. Неизвестный intent не
исполняется.

`admin-bot.yml` передаёт `natural_language_admin` feature flag и настройки
провайдера. Если `OPENAI_API_KEY` или `BBVG_AI_MODEL` отсутствуют, широкий
AI-разбор недоступен, но детерминированные команды продолжают работать.

## Тестовый контракт

- `tests/test_ai_core.py` — fallback, кэш, rate limit, structured JSON и ошибка провайдера;
- `tests/test_suspicious_posts.py` — candidate gate, отсутствие влияния на wheel state, пороги и дедупликация;
- `tests/test_source_discovery_lifecycle.py` — переходы candidate/observed/recommended/approved/rejected;
- `tests/test_hunter_profile.py` — дедупликация событий, ручные/авто участия, серии и порядок меню;
- `tests/test_natural_language_admin.py` — read-only выполнение, обязательное подтверждение записи, отказ критических операций и fallback неизвестной команды.

Production workflow дополнительно выполняют собственные compile/self-test/acceptance
контракты. Успешный коммит сам по себе не считается доказательством deploy:
проверяются heartbeat monitor и control center.
