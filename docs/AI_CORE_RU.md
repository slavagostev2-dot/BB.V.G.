# BB V.G. — AI Core и интеллектуальные функции

## Назначение

`bbvg/ai_core.py` — единый необязательный шлюз для интеллектуальных функций
проекта. Он не владеет обнаружением колёс, уведомлениями, автоучастием или
жизненным циклом событий и не является обязательной зависимостью этих контуров.

Главный принцип: отказ внешнего AI не останавливает BB V.G. При ошибке модели,
исчерпании лимита, неверном JSON или отсутствии настройки каждый предметный
модуль получает детерминированный fallback.

## Провайдеры

Поддерживаются два провайдера:

- `gemini` — основной бесплатный провайдер проекта;
- `openai` — совместимый резервный провайдер.

Выбор задаётся переменной `BBVG_AI_PROVIDER`. Модель задаётся
`BBVG_AI_MODEL`. Для Gemini ключ читается только из `GEMINI_API_KEY`, для
OpenAI — только из `OPENAI_API_KEY`. Значения ключей не записываются в runtime
state, журнал решений или сообщения.

Текущая production-конфигурация использует Gemini. Значение
`gemini-2.5-flash-lite`, ранее указанное в настройках, автоматически переводится
AI Core на актуальную стабильную модель `gemini-3.1-flash-lite`. Поэтому менять
созданную пользователем GitHub variable вручную не требуется. В журнал решений и
диагностику записывается фактически использованная модель.

Gemini вызывается через `generateContent` с ключом в заголовке
`x-goog-api-key`. Для структурированных запросов задаётся JSON MIME type.

## Feature flags

- `health_inspector` — объяснение результатов системной диагностики;
- `suspicious_post_analysis` — классификация неоднозначных Telegram-публикаций;
- `source_discovery` — дополнительный слой оценки кандидатов источников;
- `natural_language_admin` — разбор свободных администраторских команд.

Глобальное включение:

```text
BBVG_AI_ENABLED=true
```

Список функций:

```text
BBVG_AI_FEATURES=health_inspector,natural_language_admin
```

## Кэш, лимиты и журнал

`ai_runtime_state.json` хранит ограниченный журнал решений, минутный rate limit и
кэш одинаковых запросов. По умолчанию:

- не более 10 новых запросов в минуту;
- TTL кэша 900 секунд;
- не более 500 последних решений.

В журнале сохраняются feature, провайдер, фактическая модель, статус, confidence,
классификация и hash результата. Исходный prompt, API-ключи и приватное состояние
не сохраняются.

## Защита данных

Во внешний AI не передаются:

- API-ключи и GitHub secrets;
- Telegram bot token;
- зашифрованный private state;
- cookies и BetBoom storage state;
- идентификаторы пользователей, когда они не нужны предметной функции.

Анализ сомнительных публикаций получает только публичный текст поста и
минимальные публичные метаданные. Инспектор здоровья получает сокращённый
технический snapshot без секретов. Администраторские команды проходят allowlist
intent и предметную проверку прав.

## Пункт 10 — инспектор здоровья

Владелец: `bbvg/health_inspector.py`.

Строгие проверки `system_checks` определяют уровень `ok`, `warning` или
`critical`. AI не может изменить severity: он только формулирует понятное
резюме, влияние и безопасное рекомендуемое действие.

Инспектор read-only. Он не перезапускает workflow, не меняет источники и не
исправляет state самостоятельно. Результат хранится в
`system_check_state.json.ai_health_inspector`.

Production workflow передаёт:

```text
BBVG_AI_FEATURES=health_inspector
BBVG_AI_PROVIDER=${{ vars.BBVG_AI_PROVIDER }}
BBVG_AI_MODEL=${{ vars.BBVG_AI_MODEL }}
GEMINI_API_KEY=${{ secrets.GEMINI_API_KEY }}
```

Перед диагностикой workflow проверяет, что выбранный провайдер и его ключ
настроены. Живой вызов Gemini подтверждён статусом `ai_status=ok`, режимом `ai`
и `used_fallback=false`.

## Пункт 3 — анализ сомнительных публикаций

Владелец: `bbvg/monitor/suspicious_posts.py`.

Модуль рассматривает только свежие публичные публикации, в которых обычный
парсер не нашёл прямую ссылку `betboom.ru/freestream/...`. Перед AI работает
локальный candidate gate, поэтому обычные сообщения не расходуют квоту.

Допустимые классы:

- `active_wheel`;
- `possible_wheel_announcement`;
- `old_wheel_reference`;
- `betboom_promotion`;
- `irrelevant`;
- `uncertain`.

AI-классификация не создаёт активное колесо, не запускает автоучастие и не
обходит BetBoom API. При высокой уверенности она формирует только служебный
сигнал администратору. Реальное событие появляется после прямой ссылки и обычной
технической проверки BetBoom.

Monitor workflow теперь передаёт Gemini-настройки и включает:

```text
BBVG_AI_FEATURES=suspicious_post_analysis,source_discovery
```

История хранится в `source_stats.json.ai_post_analysis` без исходного текста:
только hash, источник, публичный message ID/URL, результат, confidence и ID
решения.

## Пункт 4 — поиск новых источников

Владелец: `bbvg/monitor/source_discovery.py` поверх существующей разведки.

Жизненный цикл кандидата:

```text
candidate → observed → recommended → approved/rejected
```

Базовая рекомендация остаётся детерминированной и учитывает публичность,
транспортную проверку, найденные колёса, независимые упоминания и рейтинг.
Feature `source_discovery` включён в monitor как дополнительный AI-слой, но AI
не может самостоятельно добавить канал. Решение подтверждает администратор.

## Пункт 7 — профиль охотника

Владелец: `bbvg/bot/profile.py`.

Профиль не зависит от внешнего AI. Статистика пересчитывается из реальных
событий участия и подтверждённых автоучастий владельца. Разные технические
идентификаторы одного поколения нормализуются, чтобы одно событие не считалось
дважды.

## Пункт 12 — управление естественным языком

Владелец: `bbvg/bot/natural_language_admin.py`.

Слой доступен только owner/admin. Типовые команды сначала разбираются локальными
правилами. Нестандартные формулировки передаются Gemini через allowlist intent.

Read-only команды могут выполняться сразу: активные колёса, статус, рейтинг,
источники и профиль. Изменение интервала или режима источника требует отдельного
подтверждения кнопкой. Управление секретами, передача владельца, массовое
удаление, переписывание Git-истории и удаление backup через AI запрещены.

Control Center включает:

```text
BBVG_AI_FEATURES=natural_language_admin
```

Если провайдер временно недоступен, детерминированные команды и обычная панель
продолжают работать.

## Тестовый и production-контракт

- `tests/test_ai_core.py` проверяет fallback, кэш, rate limit, structured JSON,
  Gemini header/endpoint и обработку заблокированного ответа;
- предметные тесты проверяют сомнительные посты, discovery, профиль и безопасные
  администраторские команды;
- workflow перед запуском проверяют наличие модели и ключа выбранного провайдера;
- успешным развёртывание считается только после живого provider call, свежего
  heartbeat Control Center и успешной итерации monitor.
