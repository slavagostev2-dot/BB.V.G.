# BB V.G. — инспектор здоровья

## Назначение

`bbvg/health_inspector.py` — read-only слой объяснения поверх существующей системной диагностики. Он не определяет факты самостоятельно и не выполняет эксплуатационные действия.

Источником истины остаются `system_checks_v3.py`, `system_checks.py` и `incident_manager.py`.

## Порядок работы

1. Штатные проверки формируют findings и открывают либо закрывают инциденты.
2. Инспектор рассчитывает детерминированную оценку `ok`, `warning` или `critical`.
3. В AI передаётся только сокращённый диагностический whitelist: агрегированные показатели монитора, inventory, wheel API и обезличенные findings.
4. AI может сформулировать только понятное резюме, влияние проблемы и рекомендуемое действие.
5. Итоговая severity и необходимость внимания администратора всегда берутся из детерминированных правил.
6. При недоступности AI используется тот же rules-only результат без остановки системных проверок.

## Уведомления

При новом критическом инциденте существующая единая административная сводка дополняется блоком «Инспектор здоровья». Для состояний `ok` и `warning` отдельное сообщение не создаётся. Дедупликация и уведомления о восстановлении по-прежнему принадлежат `incident_manager.py`.

## Состояние

Результат каждой системной проверки записывается в поле `ai_health_inspector` файла `system_check_state.json`.

AI-журнал и кэш используют существующий `ai_runtime_state.json`. Если AI фактически не вызывается, основная диагностика всё равно полностью работоспособна.

## Безопасность

Инспектор не получает пользовательские профили и произвольные поля системного состояния. AI-ответ не может понизить критический статус, созданный правилами, и не используется как команда для исполнения.

## Проверки

`tests/test_ai_health_inspector.py` проверяет rules-only fallback, неизменность критической severity, whitelist входных данных и отсутствие отдельного admin-note для не критических состояний.
