# План безопасного рефакторинга BB V.G.

## Цель

Привести репозиторий к структуре, в которой по имени файла и папки понятно его назначение, без цепочек `runtime_v2 ... runtime_v41`, разрозненных `chapter*`-проверок и дублирующих workflow.

Качество, обратимость и полнота проверок важнее скорости.

## Неприкосновенные требования

1. До каждого крупного этапа существует точка отката.
2. Рабочие данные пользователей, ролей, уведомлений, рейтинга и колёс не мигрируются без отдельной проверки.
3. Старый файл удаляется только после того, как:
   - на него нет импортов;
   - на него нет ссылок в GitHub Actions, тестах и preflight;
   - новый модуль покрывает его поведение;
   - тесты и сценарии проходят;
   - панель и монитор оставляют свежий heartbeat.
4. История старого кода остаётся доступной в Git и в резервной ветке.
5. Новые обновления не создают файлы с номерами версий. Версия хранится в одном месте, а код меняется в стабильных модулях.

## Целевая структура

```text
bbvg/
  bot/
    app.py                 # точка запуска Telegram-панели
    runtime.py             # цикл getUpdates и диспетчеризация
    menus.py               # главное меню и навигация
    wheels.py              # экран активных колёс и действия
    users.py               # роли, пользователи и доступ
    reports.py             # статистика и сводки
  monitor/
    app.py                 # точка запуска поиска колёс
    scanner.py             # обход Telegram-источников
    parser.py              # извлечение ссылок и времени
    lifecycle.py           # жизненный цикл и защита от дублей
    delivery.py            # уведомления и дедупликация доставки
    sources.py             # каталог и здоровье источников
  domain/
    wheels.py              # модели и правила колеса
    ratings.py             # начисление рейтинга
    users.py               # модели ролей и настроек
  storage/
    private_state.py       # зашифрованное состояние пользователей
    runtime_state.py       # состояние монитора
    repositories.py        # единый слой чтения/записи JSON
  operations/
    health.py              # heartbeat и диагностика
    incidents.py           # подавление и восстановление инцидентов
    migrations.py          # явные миграции форматов данных
  tests/
    unit/
    integration/
    scenarios/
  docs/
    ARCHITECTURE_RU.md
    CHAT_CONTEXT_RU.md
    OPERATIONS_RU.md
```

На первом этапе допускается более компактная структура, если разделение на пакеты потребует слишком рискованного переноса. Главное — уйти от файлов с версиями и закрепить понятные границы ответственности.

## Этапы

### Этап A. Инвентаризация

- построить список файлов и назначений;
- построить граф импортов;
- найти точки запуска из workflow;
- классифицировать файлы: production, tests, docs, migration, runtime data, archive;
- выделить фактическую цепочку наследования панели;
- найти неиспользуемые workflow и скрипты.

Результат: `docs/CODE_INVENTORY_RU.md`.

### Этап B. Зафиксировать контракты

Перед переносом создать тесты текущего поведения:

- главное меню и роли;
- список активных колёс и callback_data;
- сводка;
- участие, завершение, неактивное и ручное время;
- рейтинг без двойного начисления;
- повтор одной ссылки по таймеру и по правилу двух часов;
- ночной перезапуск;
- чтение кнопок Telegram-поста;
- сохранение ролей и уведомлений.

### Этап C. Единая панель

- создать стабильный модуль без номера версии;
- перенести фактическое поведение текущей v41;
- заменить последовательное наследование на композицию модулей;
- оставить временный совместимый запуск старого имени;
- переключить workflow и тесты;
- удалить старые runtime-файлы только после полного подтверждения.

### Этап D. Единые проверки

- объединить `chapter1_stability.py`, `chapter4_acceptance.py`, `chapter5_acceptance.py` и связанные проверки в `tests/production_acceptance.py`;
- сохранить отдельные тестовые функции по предметным областям;
- убрать зависимость preflight от текста `print()`;
- документацию глав объединить в архитектурный документ, сохранив полезные правила.

### Этап E. Папки монитора и состояния

- перенести код поиска, транспорта, парсера, жизненного цикла и уведомлений в понятные пакеты;
- сначала оставить совместимые файлы-переходники на старых путях;
- после замены всех импортов удалить переходники.

### Этап F. GitHub Actions

Оставить минимальный набор:

- `bot.yml` — Telegram-панель;
- `monitor.yml` — основной монитор;
- `nightly.yml` — ночное обнаружение источников;
- `health.yml` — диагностика;
- `tests.yml` — проверки pull request/ветки.

Временные recovery- и versioned-workflow удалить после подтверждения замены.

### Этап G. Финальная чистка

- удалить доказанно неиспользуемые файлы;
- проверить отсутствие импортов старых имён;
- проверить дерево репозитория;
- прогнать полный набор тестов;
- развернуть на тестовом запуске;
- проверить heartbeat панели и монитора;
- только затем переключить `main`.

## Правила именования после рефакторинга

- без суффиксов `v2`, `v41`, `new`, `final`, `fixed`;
- имя отражает ответственность: `wheel_lifecycle.py`, а не `chapter5.py`;
- один production-модуль — одна основная ответственность;
- файлы миграций именуются по формату данных, а не по версии приложения;
- runtime JSON находятся в отдельной папке данных либо имеют явный список в документации;
- временный диагностический файл обязан иметь срок удаления и отметку в плане.

## Откат

Рабочая контрольная точка перед рефакторингом:

`backup/v41-before-cleanup-2026-07-16`

При критической ошибке `main` возвращается на эту ветку/коммит. После возврата отдельно запускаются панель и монитор, затем проверяются их heartbeat.
