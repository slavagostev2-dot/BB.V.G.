# Контекст BB V.G. для нового чата

Скопируйте этот текст в новый чат перед изменениями проекта.

---

Мы работаем с репозиторием `slavagostev2-dot/betboom-wheel-monitor`.

## Назначение проекта

BB V.G. — Telegram-бот и монитор колёс BetBoom. Система:

- проверяет Telegram-каналы и чаты;
- извлекает ссылки `betboom.ru/freestream/...`;
- определяет время прокрутки;
- подавляет повторные публикации;
- уведомляет пользователей;
- хранит активные колёса и участие;
- ведёт рейтинг источников;
- предоставляет административную Telegram-панель.

## Главное правило разработки

Не создавай новые файлы вида `runtime_v42.py`, `final_fix.py`, `new_monitor.py` или `chapter6.py`.

Сначала найди существующий модуль по ответственности и измени его. Если архитектура ещё находится в переходном состоянии, проверь `docs/REFACTOR_PLAN_RU.md` и текущую рабочую ветку.

## Важные правила поведения

1. Одинаковый Telegram-пост не отправляется повторно.
2. Одинаковая ссылка из другого поста подавляется:
   - до автоматически найденного времени прокрутки;
   - до времени, установленного администратором;
   - два часа, если времени нет.
3. После окончания срока та же ссылка может создать новое событие.
4. Ручной таймер имеет приоритет над двухчасовым правилом.
5. Кнопка `Завершено` начисляет рейтинг один раз для текущего события.
6. Повторное подтверждение не удваивает рейтинг.
7. Callback-данные Telegram нельзя менять без теста совместимости и ограничения 64 байта.
8. Состояние пользователей и ролей хранится зашифрованно; миграция не должна менять владельца, администраторов и блокировки.
9. Панель — единственный потребитель `getUpdates`; монитор не должен конкурировать с ней за callback-события.
10. Нельзя считать обновление рабочим без свежего heartbeat панели и монитора.

## Где искать код после рефакторинга

Планируемая структура:

- `bbvg/bot/` — Telegram-панель, меню, пользователи, сводки и действия;
- `bbvg/monitor/` — поиск источников, парсинг постов, жизненный цикл колёс и доставка;
- `bbvg/domain/` — бизнес-правила колёс, рейтинга и пользователей;
- `bbvg/storage/` — чтение, запись и шифрование состояния;
- `bbvg/operations/` — heartbeat, диагностика, инциденты и миграции;
- `tests/` — unit, integration и сценарные проверки;
- `docs/` — архитектура и эксплуатация;
- `.github/workflows/` — только точки запуска и CI, без бизнес-логики.

Пока рефакторинг не завершён, текущие старые пути могут ещё использоваться. Не удаляй их только по названию: сначала проверь импорты, workflow, preflight и тесты.

## Перед любым изменением

1. Прочитай `docs/REFACTOR_PLAN_RU.md`.
2. Проверь текущую ветку и активную версию.
3. Найди все импорты и workflow-ссылки на изменяемый файл.
4. Зафиксируй ожидаемые сценарии тестами.
5. Вноси изменение небольшим этапом.
6. Запусти проверки.
7. Подтверди свежий heartbeat панели и/или монитора.
8. Не удаляй старый файл, пока на него есть хотя бы одна ссылка.
