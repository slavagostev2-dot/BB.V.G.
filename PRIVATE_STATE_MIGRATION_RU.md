# Перенос личного состояния в Cloudflare D1

Этот документ относится к переходу панели и Mini App на `admin_panel_runtime_v22.py` и Worker из каталога `state_api/`.

## Что выполняется автоматически после объединения PR

1. Workflow `state-api.yml` создаёт или находит D1-базу `bbvg-private-state` с юрисдикцией `eu`.
2. В базе создаются таблицы пользователей, ролей, настроек уведомлений, участий, скрытых колёс, пользовательских настроек и заявок источников.
3. Workflow ищет последнюю непустую версию `bot_access.json` и `source_requests.json` в Git-истории и переносит записи в D1. Содержимое записей в лог не выводится.
4. Worker получает `BOT_TOKEN` как закрытый Cloudflare secret и публикует HTTPS endpoint.
5. URL Worker записывается в `private_state_deployment.json` без токенов и персональных данных.
6. Перезапускаются Mini App, Telegram-панель и основной монитор.
7. Панель запускается на `v22` только после успешного развёртывания Worker.

## Проверка перед очисткой истории

Историю нельзя переписывать до подтверждения миграции. Сначала должны выполняться все условия:

- workflow `Deploy BB V.G. private state API` завершён успешно;
- `private_state_deployment.json` имеет статус `deployed` и HTTPS URL;
- существующие владелец и администраторы видны в панели;
- новый пользователь после открытия Mini App появляется в панели без отправки сообщения боту;
- отметка «Я участвую» одинакова в Mini App и боте;
- скрытое колесо не показывается этому пользователю после повторного открытия;
- заявка источника появляется у администратора, а Mini App показывает подтверждение только после ответа API.

## Очистка Git-истории

Очистка истории является отдельной разрушительной операцией: меняются SHA всех затронутых коммитов, открытые ветки и локальные клоны потребуется синхронизировать заново. Выполнять её следует только после проверки D1 и резервного экспорта базы.

На чистом локальном клоне с установленным `git-filter-repo`:

```bash
git clone --mirror https://github.com/slavagostev2-dot/betboom-wheel-monitor.git
cd betboom-wheel-monitor.git

git filter-repo \
  --path bot_access.json \
  --path source_requests.json \
  --invert-paths \
  --force

git push --force --mirror
```

После переписывания истории нужно в обычном клоне текущего `main` заново добавить обезличенные шаблоны `bot_access.json` и `source_requests.json`, проверить обязательные workflow и сообщить участникам проекта, что старые локальные клоны следует удалить или полностью пересоздать.

## Ограничение

Удаление файлов из текущей ветки не удаляет их из уже созданных форков, клонов, архивов и внешних кэшей. История основного репозитория очищается только отдельной операцией выше.
