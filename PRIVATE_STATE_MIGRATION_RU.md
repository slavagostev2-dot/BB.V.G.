# Архив: неиспользованный перенос личного состояния в Cloudflare D1

Этот документ сохраняет историю отменённого решения. Ветка панели `admin_panel_runtime_v23.py`, Worker из `state_api/` и D1-база **не входят в действующую архитектуру BB V.G.**

## Почему решение не используется

Эксперимент предполагал отдельный Worker, D1-базу, дополнительный API и отдельную ветку runtime панели. Он увеличивал количество точек отказа, усложнял миграцию ролей и создавал параллельный источник состояния.

Текущая production-система использует:

- зашифрованный файл `bot_private_state.enc.json`;
- модуль `bot_private_state.py`;
- миграцию `migrate_bot_private_state.py`;
- трёхстороннее слияние ролей и пользователей в действующей цепочке панели;
- отдельный ключ `BOT_STATE_KEY`;
- проверку сохранности ролей перед каждым запуском панели.

Источником истины для дальнейшего рефакторинга являются `AGENTS.md`, `docs/CODE_INVENTORY_RU.md` и `docs/PROJECT_CHANGELOG_RU.md`.

## Исторический план D1

Отменённый вариант предполагал:

1. создание D1-базы `bbvg-private-state`;
2. развёртывание Worker с закрытым `BOT_TOKEN`;
3. перенос пользователей, ролей, настроек, участий и заявок;
4. запуск отдельной панели v23 только после статуса `deployed`;
5. последующее переписывание Git-истории.

Эта последовательность не должна запускаться без нового отдельного архитектурного решения владельца проекта.

## Очистка Git-истории

Переписывание истории остаётся отдельной разрушительной операцией. Оно меняет SHA коммитов, требует повторной синхронизации веток и не удаляет данные из уже созданных форков, архивов, клонов и внешних кэшей.

В рамках текущего рефакторинга история Git не переписывается. Устаревшие файлы удаляются только из актуальной ветки после переноса полезной логики и полного тестового прогона; их прежние версии остаются доступными в истории и резервной ветке.
